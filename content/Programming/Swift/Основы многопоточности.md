Мы узнали о потоках и как они помогают нам выполнять задачи в фоне. Но что случится, если эти потоки попытаются одновременно работать с одними и теми же данными? Тут может возникнуть такая неприятная вещь, как "состояние гонки", которое способно вызвать множество проблем.

В этом уроке мы подробнее разберемся, что такое это "состояние гонки", поймем, как оно может все перевернуть вверх дном, и как нам не дать этому случиться, используя такие полезные инструменты синхронизации, как блокировки и семафоры.

## Что такое состояние гонки?

Состояние гонки — это ошибка. Она возникает в многопоточной программе, когда результат выполнения этой программы определяется тем, в какой последовательности выполнятся фрагменты вычислений. Представьте себе два потока, которые одновременно пытаются увеличить один и тот же счетчик. Оба потока считывают значение счетчика, увеличивают его и сохраняют новое значение. Но поскольку эти потоки не синхронизированы, они могут перезаписывать изменения друг друга, и итоговое значение счетчика может оказаться неверным. Это и есть состояние гонки, и оно может привести к повреждению данных или даже к сбоям.

## Как происходит состояние гонки?

Давайте рассмотрим простой пример. У нас есть счетчик:

```swift
var counter = 0

```

И две функции, которые увеличивают этот счетчик:

```swift
func incrementCounter() {
  counter += 1
}

func incrementCounterAgain() {
  counter += 1
}

```

Мы вызываем эти функции из разных потоков:

```swift
let queue1 = DispatchQueue(label: "serial")
let queue2 = DispatchQueue(label: "serial")

queue1.async {
  incrementCounter()
}

queue2.async {
  incrementCounterAgain()
}

```

Мы ожидаем, что окончательное значение счетчика будет равно 2. Однако, поскольку потоки не синхронизированы, фактический результат может быть 1 или 2, в зависимости от порядка, в котором выполняются потоки. Этот непредсказуемый результат является состоянием гонки.

## Синхронизация потоков

Как же мы можем предотвратить состояние гонки? В этом нам поможет такой концепт как синхронизация потоков.

Чтобы предотвратить состояние гонки, мы можем использовать инструменты синхронизации, чтобы гарантировать, что только один поток может получить доступ к данным в определенный момент времени. Это похоже на призыв "По одному, пожалуйста!".

В Swift доступны несколько инструментов для синхронизации потоков, включая блокировки и семафоры.

## Блокировки

Блокировка - это как билет на аттракцион в парке развлечений, который может использовать только один человек за раз. Аналогично, только один поток может "захватывать" блокировку в определенный момент времени.

Блокировку можно использовать следующим образом:

```swift
let lock = NSLock()

lock.lock()
// получить доступ к общему ресурсу
lock.unlock()

```

Когда поток встречает `lock.lock()`, он проверяет, доступна ли блокировка. Если да, поток захватывает блокировку и продолжает работу. Если нет, поток ждет, пока блокировка не будет освобождена с помощью `lock.unlock()`.

Для нашего примера со счетчиком мы можем использовать блокировку, чтобы предотвратить состояние гонки:

```swift
let lock = NSLock()
var counter = 0

func incrementCounter() {
  lock.lock()
  counter += 1
  lock.unlock()
}

```

Теперь только один поток может увеличивать счетчик в одно и то же время, что предотвращает состояние гонки.

### Блокировки в GCD

GCD (Grand Central Dispatch) также использует блокировки на низком уровне для обеспечения согласованности при выполнении задач в различных очередях. Каждая очередь GCD ассоциируется с блокировкой, которую поток должен захватить, чтобы начать выполнение задачи в этой очереди. Только одна задача может удерживать блокировку очереди в определенный момент времени, что обеспечивает взаимное исключение.

Например, предположим, у нас есть конкурентная очередь:

```swift
let queue = DispatchQueue(label: "concurrent", attributes: .concurrent)

```

Эта очередь позволяет выполнять несколько задач одновременно (параллельно). Но как она предотвращает те ситуации, когда задачи мешают друг другу?

"Под капотом" GCD связывает блокировку с каждой очередью. Когда задача начинает выполнение, она сначала получает блокировку, подобно билету на аттракцион. Пока задача удерживает блокировку, никакая другая задача не может начать выполнение. По мере завершения задачи, она освобождает блокировку. Это позволяет следующей задаче в очереди получить блокировку и начать выполнение.

Основное отличие между GCD и обычными блокировками заключается в том, что управление блокировками автоматически обрабатывается фреймворком.

Таким образом блокировки позволяют GCD управлять доступом к очередям и синхронизировать задачи. Разработчику обычно не нужно явно использовать блокировки при работе с GCD - достаточно добавлять задачи в очереди, а синхронизацией займётся сам фреймворк.

## Остерегайтесь тупиков (Deadlocks)

Давайте поговорим о потенциальной опасности при использовании блокировок - ситуации тупика или deadlock.

Представьте, что два потока взяли по блокировке и теперь ждут, когда другой поток освободит свою. Получается замкнутый круг - каждый поток ждёт действий другого потока. Это и есть тупик. Тупики могут полностью остановить работу приложения. Как их избежать?

Важно следить за порядком получения и освобождения блокировок. Если два потока берут несколько блокировок, пусть делают это в одном и том же порядке. Тогда потоки не будут застревать в ожидании друг друга. Понимание природы тупиков и умение их избегать - ключевые навыки при работе с многопоточностью.

Нужно стараться быть осторожным и не допускать ситуаций, когда потоки бесконечно ждут друг друга! Это важно для стабильной работы приложений. Мы обсудим тупики подробнее в этой главе.

## Семафоры

Семафор похож на фейсконтроль в клубе. Он позволяет одновременный доступ к ресурсу только определенному количеству потоков.

Они полезны для предотвращения тупиков. В отличие от простых блокировок, неосторожное использование которых может привести к тупикам, семафоры позволяют более гибко ограничивать одновременный доступ к ресурсам и избегать ситуаций, когда потоки бесконечно блокируют друг друга.

Значение семафора представляет количество потоков, которые могут получить доступ к общему ресурсу одновременно.

Когда поток хочет получить доступ к ресурсу, он вызывает `semaphore.wait()`, который уменьшает значение семафора на 1. Если значение больше 0, поток может получить доступ к ресурсу. Если значение равно 0, поток блокируется и ждет, пока другой поток не вызовет `semaphore.signal()`, чтобы увеличить значение.

Например, если мы создаем семафор с начальным значением 1:

```swift
let semaphore = DispatchSemaphore(value: 1)

semaphore.wait()
// получить доступ к ресурсу
semaphore.signal()

```

Это означает, что только 1 поток может получить доступ к ресурсу за один раз. Когда поток закончил работу с ресурсом, он вызывает `semaphore.signal()`, чтобы увеличить значение обратно до 1, разрешая другому ожидающему потоку получить доступ.

Таким образом, значение представляет текущее доступное "место" для потоков, чтобы получить одновременный доступ к ресурсу. Инициализируя семафор с разными значениями, вы можете контролировать количество потоков с одновременным доступом.

Мы можем использовать семафор, чтобы ограничить доступ к нашему счетчику:

```swift
let semaphore = DispatchSemaphore(value: 1)

func incrementCounter() {
  semaphore.wait()
  counter += 1
  semaphore.signal()
}

```

В данном примере только один поток может увеличивать счетчик в одно и то же время, что опять же предотвращает состояние гонки.

## Барьеры в Grand Central Dispatch

Grand Central Dispatch предоставляет функционал барьеров - это специальные задачи, которые помогают координировать доступ к общим ресурсам в многопоточных приложениях. Барьер выполняется в "изоляции" - он ждет, пока все предыдущие задачи в очереди завершатся, и блокирует выполнение последующих задач, пока сам не закончит работу. Это особенно полезно, когда нужно атомарно (то есть неделимо, как атом) записать данные в общий ресурс. Барьер гарантирует, что никакая другая задача не сможет одновременно читать или записывать в этот ресурс. А это как раз то, что нам нужно!

Вы можете добавить задачу-барьер в очередь, используя методы `async(flags:execute:)` или `sync(flags:execute:)` вашей очереди dispatch и передавая флаг `.barrier`. Вот пример:

```swift
let concurrentQueue = DispatchQueue(label: "com.example.queue", attributes: .concurrent)

concurrentQueue.async(flags: .barrier) {
    // Запись в общий ресурс
}

```

В приведенном выше примере задача, помеченная флагом `.barrier`, будет выполняться только после завершения всех задач, добавленных в `concurrentQueue` перед ней, и она не позволит выполниться другим задачам, пока она сама не завершится.

Важно отметить, что преграды работают как ожидается только с параллельными очередями, которые вы создали самостоятельно. Они не будут работать с глобальными параллельными очередями (о них поговорим позже), предоставляемыми системой, потому что GCD не гарантирует, что он будет ждать завершения всех задач перед выполнением преграды.

Давайте посмотрим на некоторые примеры с операторами print, чтобы иллюстрировать порядок выполнения и то, как флаг GCD `.barrier` может его изменить.

### Пример 1: Без преграды

В следующем примере мы будем использовать параллельную очередь dispatch для выполнения нескольких задач. Без использования флага `.barrier`.

```swift
let concurrentQueue = DispatchQueue(label: "com.example.queue", attributes: .concurrent)

concurrentQueue.async {
    print("Задача 1 началась")
    sleep(2)
    print("Задача 1 завершена")
}

concurrentQueue.async {
    print("Задача 2 началась")
    sleep(1)
    print("Задача 2 завершена")
}

concurrentQueue.async {
    print("Задача 3 началась")
    sleep(1)
    print("Задача 3 завершена")
}

```

Поскольку мы используем параллельную очередь, задачи начинаются как можно скорее, и они не обязательно заканчиваются в том порядке, в котором они были добавлены. Задача 2 и Задача 3 могут завершиться раньше Задачи 1 из-за их более короткого времени ожидания.

### Пример 2: С барьером

Давайте посмотрим на пример, когда добавляется задача с барьером в ту же очередь:

```swift
concurrentQueue.async(flags: .barrier) {
    print("Началась задача-преграда")
    sleep(1)
    print("Задача-преграда завершена")
}

concurrentQueue.async {
    print("Задача 4 началась")
    sleep(1)
    print("Задача 4 завершена")
}

```

В этом случае задача-барьер будет ждать, пока Задача 1, Задача 2 и Задача 3 не завершатся, прежде чем она начнется. Как только задача-преграда начнется, она завершится до начала Задачи 4. Несмотря на то, что Задача 4 добавлена в очередь до завершения задачи-барьера, она не начнется до завершения задачи-барьера.

Используя барьер, мы можем контролировать порядок выполнения, гарантируя, что определенные задачи не начнутся, пока другие не завершатся.

Это может быть особенно полезно, когда у вас есть общий ресурс, который вы хотите обновить (записать) без одновременного чтения или записи другими задачами.

### Краткое содержание

- Состояния гонки возникают, когда потоки одновременно обращаются к общим ресурсам, что приводит к непредсказуемым результатам. Это общая проблема в многопоточном программировании.
- Инструменты синхронизации, такие как блокировки, семафоры и барьеры, помогают предотвратить состояния гонки, контролируя доступ потоков к общим ресурсам.
- Взаимные блокировки могут возникнуть, когда два или более потока ждут, пока друг друга освободят ресурс, что приводит к остановке приложения. Очень важно тщательно использовать инструменты синхронизации, чтобы избежать взаимных блокировок.
- Функциональность барьера GCD позволяет контролировать параллельное выполнение задач в очереди диспетчеризации. Это полезный инструмент для предотвращения состояний гонки при работе с общими ресурсами.

Управление параллельностью в ваших iOS-приложениях - это важный навык. С глубоким пониманием потоков, состояний гонки и инструментов синхронизации вы уже находитесь на пути к написанию надежных и эффективных многопоточных приложений.

### 2. Задание по программированию

Реализуйте потокобезопасный стек с помощью GCD. Стек должен поддерживать методы `push()` и `pop()`, которые добавляют/удаляют элементы безопасным для потоков способом.

```swift
class ThreadSafeStack {
  private var items = [Int]()

  func push(_ item: Int) {
    // Добавьте код здесь
  }

  func pop() -> Int? {
    // Добавьте код здесь
  }
}

let stack = ThreadSafeStack()
let queue1 = DispatchQueue(label: "serial1")
let queue2 = DispatchQueue(label: "serial2")
let queue3 = DispatchQueue(label: "serial3")

queue1.async {
  stack.push(1)
}

queue2.async {
  stack.push(2)
}

queue3.async {
  stack.push(3)
}

let queue4 = DispatchQueue(label: "concurrent", attributes: .concurrent)
    queue4.async {
    // Вывод может меняться в зависимости от последовательности выполнения
    // Здесь основное внимание уделяется обеспечению потокобезопасности операций стека, а не порядка вывода
      for _ in 1...3 {
        if let item = stack.pop() {
          print(item)
        }
      }
    }

```

Вот реализация с использованием DispatchQueue GCD для обеспечения безопасности потоков:

```swift
class ThreadSafeStack {

  private let stackQueue = DispatchQueue(label: "stack")

  private var items = [Int]()

  func push(_ item: Int) {
    stackQueue.async {
      self.items.append(item)
    }
  }

  func pop() -> Int? {
    var result: Int?

    stackQueue.sync {
      result = self.items.popLast()
    }

    return result
  }
}

```

### 3. Задача со звездочкой

Добавьте новый метод `peek()`, который возвращает верхний элемент, не удаляя его. Как вы можете реализовать это безопасным для потоков способом?

_Образец решения:_

```swift
  func peek() -> Int? {
    var result: Int?
    stackQueue.sync {
      result = self.items.last
    }
    return result
  }

```

Метод `peek()` использует такой же подход к синхронизации, как `pop()` - он оборачивает чтение стека в блоке синхронизации в очереди, чтобы обеспечить исключительный доступ. Это позволяет ему безопасно считывать последний элемент без его удаления.

# Подведение итогов

- Состояния гонки возникают, когда потоки одновременно обращаются к общим данным и вызывают непредвиденное поведение.
- Инструменты, такие как блокировки, семафоры и барьеры, синхронизируют выполнение потоков и предотвращают состояния гонки.
- Блокировки предоставляют доступ только одному потоку за раз. Семафоры предоставляют доступ заданному количеству потоков.
- Получение и освобождение блокировок в разных порядках может вызвать взаимную блокировку.
- Используйте барьеры GCD для изолирования выполнения кода в разных потоках при необходимости.
- Практикуйтесь в использовании техник синхронизации, таких как блокировки и семафоры, в образцовых приложениях.
- Учитеcь выявлять участки кода, подверженные состояниям гонки, при работе с многопоточным кодом.
